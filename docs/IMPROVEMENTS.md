# 今後の改善方針

## より厳密な監視キー
- 現状は通知マッチングを `planId + mealText` に緩めています（`roomClass/roomName` の日付揺れを回避するため）。
- 将来的には、希望日での「正しい room 情報」を保存し、`planId + roomId（可能なら）+ mealText` で監視するよう改善したいです。

## 予約URLの整形
- 通知内の「予約: `RsvInput.do`」URLは、VacantHotelSearchの `reserveUrl` のクエリを流用して組み立てています。
- 追加で、URLパラメータが不足している場合のフォールバック（またはAPIレスポンスから必要値をより確実に抽出）を検討します。

## UI/UX
- 「監視に追加」時に保存内容をプレビュー表示し、「どの日付でマッチする前提か」を説明する表示を追加したいです。

## 429対策の強化
- 固定日や希望日を増やすと429が出やすくなります。
- 外側リトライだけでなく、待ち時間・リクエスト間隔を状況に応じて動的に調整する方向を検討します。
